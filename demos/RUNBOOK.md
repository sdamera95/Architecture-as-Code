# Demos Runbook — Ubuntu 24.04 / ROS2 Jazzy Workstation

All ROS2 execution happens on the remote workstation (Ubuntu 24.04, ROS2 Jazzy,
RTX PRO 6000, Isaac Sim). The macOS side only authors models, runs the bridge
pipeline, and commits the generated packages — it never builds or runs ROS2.

The committed packages (regenerated by `uv run python bridge/regen_demos.py`):

| Demo | Model | Package | Workspace path |
| --- | --- | --- | --- |
| AGR (Python) | [demos/agr/showcase_agr_full.sysml](agr/showcase_agr_full.sysml) | `autonomous_ground_robot` | `ros2-py-outputs/agr/` |
| AGR (C++) | same model | `autonomous_ground_robot` | `ros2-cpp-outputs/agr/` |
| Nav2 hybrid (Python) | [demos/agr-nav2/showcase_agr_nav2_full.sysml](agr-nav2/showcase_agr_nav2_full.sysml) | `ground_robot_with_nav2` | `ros2-py-outputs/agr-nav2/` |
| Nav2 hybrid (C++) | same model | `ground_robot_with_nav2` | `ros2-cpp-outputs/agr-nav2/` |

The Python and C++ packages share the same package *name* — build them in
separate workspaces (or `--packages-select` one at a time), not side by side.

## One-time setup

```bash
git clone https://github.com/sdamera95/Architecture-as-Code.git
cd Architecture-as-Code

# Workspace per language (the py and cpp packages share names)
mkdir -p ~/ws_py/src ~/ws_cpp/src
ln -s "$(pwd)/ros2-py-outputs"  ~/ws_py/src/demos
ln -s "$(pwd)/ros2-cpp-outputs" ~/ws_cpp/src/demos

sudo rosdep init 2>/dev/null; rosdep update
rosdep install --from-paths ~/ws_py/src ~/ws_cpp/src --ignore-src -y
# Nav2 demo additionally needs: ros-jazzy-nav2-bringup ros-jazzy-nav2-lifecycle-manager
```

## Per-iteration loop

```bash
cd Architecture-as-Code && git pull

source /opt/ros/jazzy/setup.bash
cd ~/ws_py  && colcon build                    # or: cd ~/ws_cpp
source install/setup.bash

ros2 launch ground_robot_with_nav2 ground_robot_with_nav2.launch.py
# nodes autostart via the bundled lifecycle manager (bond_timeout 0.0);
# manual alternative: ros2 run ground_robot_with_nav2 activate_nodes
```

### Verify

```bash
ros2 topic echo /conformance_report            # expect node/topic/connection/QoS OK
ros2 topic hz /scan                            # wired mode: ~10 Hz (sensor_data QoS)
ros2 topic hz /odom                            # wired mode: ~1 Hz
ros2 topic info -v /scan                       # BEST_EFFORT/VOLATILE from SensorDataQoS
ros2 lifecycle set /sensors/lidar deactivate   # timers stop; activate resumes
rqt_graph                                      # py and cpp graphs must be identical
```

The same conformance monitor script ships in both packages (byte-identical);
diffing `/conformance_report` between the py and cpp runs is the cross-language
parity check.

## Implementing demo logic (generation gap)

Hand-written behavior goes ONLY in the derived files — they are generated once
and never overwritten by regeneration:

- Python: `<pkg>/<node>_node.py` — override `handle_<port>(msg)` and the
  `on_configure_hook / on_activate_hook / on_deactivate_hook` methods.
- C++: `src/<node>_node.cpp` — same overrides; publishers are protected
  members (`posePub_->publish(msg);`).

Never edit `*_node_base.py` / `*_base.hpp` / `*_base.cpp`: those are rewritten
on every `regen_demos.py` run. Commit your derived-file edits; the macOS side
regenerates wiring around them (`--force` is the only thing that resets them).

## Isaac Sim in the loop (Nav2 demo)

Architecture reference: [agr-nav2/showcase_agr_nav2_sim.sysml](agr-nav2/showcase_agr_nav2_sim.sysml)
— the simulator and workstation are modeled as plain SysML parts the extractor
deliberately ignores (verified by `tests/test_sim_invariance.py`).

1. Start Isaac Sim with the ROS2 bridge extension enabled; map the bridge's
   topics onto the modeled names:

   | Isaac Sim source | Modeled topic |
   | --- | --- |
   | RTX lidar scan | `/scan` |
   | Depth camera point cloud / image / info | `/camera/depth/...` per model |
   | IMU sensor | `/imu/data` |
   | Differential base odometry | `/odom` |
   | Velocity command (subscriber) | `cmd_vel` chain (smoother output) |
   | Simulation clock | `/clock` |

2. Launch the stack against sim time:

   ```bash
   ros2 launch ground_robot_with_nav2 ground_robot_with_nav2.launch.py \
       use_sim_time:=true
   ```

3. Tell the monitor which out-of-model nodes are tolerated (Isaac bridge,
   rviz, rqt). Without this, the monitor reports them at WARN level (never
   ERROR) under `out_of_model`:

   ```bash
   ros2 param set /conformance_monitor ignored_node_patterns \
       "['/isaac*', '/rviz*', '/rqt*', '/transform_listener*']"
   ```

   (Or set `conformance_monitor.ros__parameters.ignored_node_patterns` in
   `config/params.yaml`.)

4. The sensor-driver nodes still run in wired mode unless their derived files
   implement real logic; for a pure sim-fed run, deactivate the drivers via
   lifecycle and let Isaac Sim's topics feed the stack:
   `ros2 lifecycle set /sensors/lidar deactivate` (etc.). The monitor's
   topic/connection checks stay green either way — topic names and types are
   what the model asserts.

## Troubleshooting

- `colcon build` fails on `nav2_lifecycle_manager`: install the Nav2 packages
  listed in one-time setup.
- Monitor reports missing nodes right after launch: lifecycle bring-up takes a
  few seconds; the report refreshes every 5 s.
- Both workspaces sourced simultaneously → package shadowing; source exactly
  one `install/setup.bash` per shell.
